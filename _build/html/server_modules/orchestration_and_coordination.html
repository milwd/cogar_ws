Orchestration And Coordination

orchestration_and_coordination.py

Order queue + state gateway for a fleet of sushi-waiter robots

The node owns a persistent FIFO of customer orders and exposes two ROS services:

  • /robot_state_decision – given the current state of a robot (Free, Busy, Wait …) it returns the next task and updates the state machine.

  • /robot_state_decision_add – appends a freshly verified customer order to the queue.

All orders survive restarts thanks to a YAML file on disk.

Persistent storage

Orders are stored in tiago_data.yaml next to this script:

orders:
  - id_client: 7
    food_list: ['sushi', 'pasta']
  - id_client: 9
    food_list: ['ramen']

The file is loaded on every service call so multiple orchestrators (or manual edits) stay in sync.

Decision rules

  1. If the queue is empty → return state_output = "Wait" and success = False.

  2. If state_input is Free or Wait and an order exists → pop the first order, return it with state_output = "Busy" and success = True.

  3. Otherwise echo the original state and set success = False.
