Orchestration & Management

orchestration_and_coordination.py

Order queue + state gateway for a fleet of sushi-waiter robots

The node owns a persistent FIFO of customer orders and exposes two ROS services:

  • /robot_state_decision – given the current state of a robot (Free, Busy, Wait …) it returns the next task and updates the state machine.

  • /robot_state_decision_add – appends a freshly verified customer order to the queue.

All orders survive restarts thanks to a YAML file on disk.

Persistent storage

Orders are stored in tiago_data.yaml next to this script:

orders:
  - id_client: 7
    food_list: ['sushi', 'pasta']
  - id_client: 9
    food_list: ['ramen']

The file is loaded on every service call so multiple orchestrators (or manual edits) stay in sync.

Decision rules

  1. If the queue is empty → return state_output = "Wait" and success = False.

  2. If state_input is Free or Wait and an order exists → pop the first order, return it with state_output = "Busy" and success = True.

  3. Otherwise echo the original state and set success = False.

task_manager.py

Central state orchestrator – queries the decision service and drives the task loop

TaskManager listens to two event streams:

  1. Action feedback (/feedback_acion) – one-line summaries published by the low-level controllers (navigation finished, manipulation failed …).

  2. Verified orders (/verif_T_manager) – customer requests already checked by the order-verification node.

Every time a new feedback line arrives the manager calls the /robot_state_decision service, handing over the current state string. The service replies with the next action the robot should perform, enabling a clean separation between decision logic (in the service) and event aggregation (in this node).

Execution loop

On every feedback update

  1. Cache the new state string.

  2. Call the decision service with that state.

  3. Log the returned instruction (no further dispatching in this stub).

Extend change_state() if you want to forward the decision to other nodes or to trigger timers / retries.
